Livery turns small programs into validated, responsive technical visuals. The same readable language works for people writing a figure and agents generating one from an explanation.

The important word there is **validated**. A renderer should not smile politely and draw a broken diagram because the source was almost correct. Livery compiles the program, solves the layout, checks the geometry, and only then hands a scene to the renderer.

```text
Livery source
  → parse and expand
  → solve the pinboard
  → validate geometry and resources
  → render or export a BoardScene
```

If that pipeline fails, you get typed diagnostics instead of a best-effort mess.

<Callout type="info" title="Working from source">
  The packages are not published to npm yet. The [installation guide](/docs/start/install) explains how to build Livery from the repository.
</Callout>

## A small example

```text
figure request("Request path") {
  client = person("Customer")
  api = service("API")
  database = database("Orders")

  send = client.right -> api.left("request")
  persist = api.right -> database.left("write", tone: success)

  row(client, api, database, gap: xl)
}
```

There are no board-level coordinates here. The bindings give every element a stable identity, the anchors make the connections explicit, and `row` gives the solver the composition rather than a pile of pixels.

## What Livery is for

Livery is aimed at technical visuals inside agent products: request paths, tool traces, data pipelines, architecture figures, state transitions, and small explainers. It gives an author enough freedom to explain visually without exposing arbitrary JavaScript or an infinite board.

That trade is deliberate. Livery is not a general illustration tool. Every accepted scene must have valid geometry.
