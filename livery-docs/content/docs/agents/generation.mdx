---
title: Generating good source
description: Structure model requests so the output stays small, valid, and visually useful.
---

The model does not need the entire compiler manual. It needs a strict output contract, the current language guide, and a visual objective that is specific enough to draw.

## Build the system instruction

```ts
import { createAgentGuide } from "@jerkeyray/core";

const system = [
  "You turn technical explanations into small Livery figures.",
  createAgentGuide({ mode: "compact" }),
].join("\n\n");
```

Keep product-specific instructions above the generated guide. Do not modify the guide to add unsupported syntax; that only teaches the model to fight the compiler.

## Write a useful user request

A good visual request contains four things:

1. **Subject** — what system or mechanism is being explained
2. **Important entities** — the parts that must appear
3. **Relationship** — sequence, hierarchy, flow, comparison, or state change
4. **Constraint** — density, viewport, or required emphasis

```text
Create one compact figure for a checkout request.

Include the customer, checkout API, payment provider, and orders database.
Show the request moving left to right. Add a timeline for request,
authorization, and completion. Keep labels under three words.
```

That is better than “make a diagram of checkout.” The model cannot read your mind, and the solver should not have to clean up an idea that never became specific.

## Ask for one figure

One generation call should produce one visual idea. If a response needs an architecture diagram, a retry timeline, and a data transformation chart, make three calls.

Large prompts encourage large scenes. Large scenes create smaller labels, more routing pressure, and worse explanations. Split by meaning before you split by viewport.

## Keep labels boring

Labels are not prose. Use `Checkout API`, not `The checkout API receives and validates the incoming request`.

Put explanation around the visual in your application. The visual should carry structure, not every sentence from the answer.

## Use color with intent

Ask for color by meaning or stable category, not as generic decoration:

```text
Make Stripe purple and Postgres blue. Use coordinated soft fills, borders,
text, and icons. Keep the remaining services neutral.
```

This gives the model an explicit visual job. Prefer semantic tones for status and exact paint overrides for brands or categories. A useful default limit is one accent family plus semantic state colors.

## Prefer structure over coordinates

Tell the model the intended flow:

```text
Place the client, gateway, worker, and database in a left-to-right sequence.
```

Do not tell it to place the gateway at `(320, 80)`. Board-level coordinates make the source brittle and throw away the point of a responsive solver.

Coordinates are appropriate only inside a bounded `canvas` for plots, mechanisms, and local illustrations.

## Use timelines only when state matters

A timeline is useful when the explanation changes:

- request → authorization → complete
- cache miss → origin read → cache fill
- retry → backoff → recovery

It is not useful as decoration on a static architecture figure. If every state shows the same thing, remove the timeline.

## Stream without flashing broken source

Token streams are incomplete by definition. Render them with `LiveryChatVisual`, pass the provider's streaming state, and let the component retain the last valid scene:

```tsx
<LiveryChatVisual
  source={accumulatedSource}
  streaming={!responseFinished}
  fallback={<p>No valid visual was produced.</p>}
/>
```

Do not show a compiler error to the user because token 143 arrived before the closing brace in token 144.

## Output handling

The guide asks for plain source without fences. Enforce that contract at your model boundary when the provider supports constrained text output.

Still compile every response. A clean-looking string is not proof that its bindings, properties, or timeline targets exist.

## A complete request

```ts
const prompt = {
  system: [
    "You create concise technical figures for an engineering assistant.",
    createAgentGuide({ mode: "compact" }),
  ].join("\n\n"),
  user: `Create one figure showing a research agent querying search,
receiving evidence, and passing it to a reasoning model.
Use a left-to-right layout and labels under three words.`,
};
```

The result should be source you can compile immediately, not a conversation about source.
