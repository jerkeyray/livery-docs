For architecture, process, and explainer diagrams, an agent should propose a typed semantic plan. Livery validates the plan, materializes responsive components and layout, and returns canonical editable source. Specialized visuals can still use the textual DSL.

```text
user intent
  → model + visualPlanSchema
  → typed nodes, edges, groups, and annotations
  → deterministic Livery source
  → compile, solve, validate
  → accept | structured diagnostic
```

```ts
import { renderVisualPlan, visualPlanSchema } from "liveryscript";

const plan = visualPlanSchema.parse(modelOutput);
const result = renderVisualPlan(plan, { width: 720 });
```

Facts such as capacities, rates, protocols, and response codes belong in plan annotations rather than artificial nodes. Use the DSL contract below for custom components, canvases, timelines, sequences, schemas, state models, and other specialized authoring.

## DSL generation: do not maintain the grammar by hand

```ts
import { createAgentGuide } from "liveryscript";

const compact = createAgentGuide({ mode: "compact" });
const reference = createAgentGuide({ mode: "reference" });
```

Compact mode belongs in normal generation and repair calls. Reference mode is for retrieval, tool help, offline agents, or debugging unfamiliar syntax.

Livery does not call a model and does not need provider keys. Your application owns the provider; Livery provides the language contract, validation, and repairs.
