---
title: Visual families
description: Outcome coverage, native kernels, and delivery status for Livery v1.
---

Livery has one composable language. A visual family supplies domain semantics and selects one of six owned layout kernels; it does not switch the parser into a separate diagram language.

The compiler catalog is the source of truth. Current families are grouped by status:

- **Supported:** flow, system context, block systems, architecture, and tree views.
- **Foundation:** interaction narratives, class models, state models, entity models, requirement models, and mindmaps.
- **Planned:** swimlanes, journeys, schedules, revision histories, timelines, kanban, event models, quantitative plots, and bounded strategic/editorial forms.

Foundation means the typed values and semantic components exist, but the complete dense/narrow/adversarial fixture gate has not passed yet. Studio rejects a generic card diagram when a request requires typed entities, states, requirements, or ordered participant messages.

The complete outcome-based inventory lives with the compiler documentation and is frozen against the Mermaid 11.16.0 public capability checklist. Mermaid source, grammar, renderers, layouts, fixtures, and visual defaults are not accepted or translated.

Every family must demonstrate original syntax and fixtures, responsive snapshots at 360–1200px, deterministic geometry, accessible reading order, and renderer parity before becoming supported.
